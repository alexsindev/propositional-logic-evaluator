## Overview

The task is to build an evaluator for propositional logic expressions that
supports two operators:

| Symbol | Meaning           | Precedence |
| ------ | ----------------- | ---------- |
| `∧`    | AND (conjunction) | Higher     |
| `∨`    | OR (disjunction)  | Lower      |

The evaluator must produce **two outputs** from a single input expression:

1. A **truth value** (`t` or `f`)
2. An equivalent expression in **prefix notation**

---

## Grammar

The formal grammar of the language is:

```
statement  ::= expr

expr       ::= expr '∨' expr       (left-associative, lower precedence)
             | expr '∧' expr       (left-associative, higher precedence)
             | 't'
             | 'f'
```

The atoms are:

- `t` — the boolean literal _true_
- `f` — the boolean literal _false_

Precedence rule (matching the assignment specification):

> `∧` binds tighter than `∨`, so `t ∨ f ∧ f` is parsed as `t ∨ (f ∧ f)`, which
> evaluates to `t`.

---

## Type of Parser

The parser is an **LALR(1)** parser, generated by the `sly` library's `Parser`
base class.

**Why LALR(1)?**

- The grammar is unambiguous when operator precedence is declared explicitly
  (which `sly` supports via its `precedence` tuple).
- LALR(1) parsers are efficient (linear time) and sufficient for this simple
  grammar.
- It fits naturally into the existing codebase, which already uses `sly` for its
  arithmetic evaluator example.

The precedence declaration inside `PropLogicParser`:

```python
precedence = (
    ("left", OR),   # lower  — ∨
    ("left", AND),  # higher — ∧
)
```

Entries listed **later** have **higher** precedence. This single declaration
handles all associativity and priority conflicts automatically — no manual
grammar rewriting needed.

| Layer              | File                           | Responsibility                                 |

| Presentation       | `main.py`                      | GUI, wires everything together                 |

## Step-by-Step Implementation

### Step 1 — AST Nodes

**File:** `src/prop_evaluator/components/ast/prop_logic.py`

**Thinking:** The assignment requires two different outputs from the same parsed
expression. If we evaluate immediately during parsing (as `MyParser` in the
example does), we lose the structure needed to produce prefix notation. Instead,
we build an **Abstract Syntax Tree** during parsing and defer all computation.

This is the same reasoning that motivated `ASTParser` in the starter project.

**Design:**

- `PropExpression` — abstract base class with two abstract methods:
  - `run() -> bool` — evaluates the expression recursively
  - `prefix() -> str` — produces the prefix-notation string recursively

- `PropTrue` / `PropFalse` — leaf nodes, no children:

  ```python
  PropTrue().run()    # → True
  PropTrue().prefix() # → "t"
  ```

- `PropAnd(left, right)` / `PropOr(left, right)` — internal nodes with two
  children:
  ```python
  PropAnd(PropFalse(), PropFalse()).run()    # → False
  PropAnd(PropFalse(), PropFalse()).prefix() # → "∧ f f"
  ```

Both `run()` and `prefix()` work by **recursively** calling the same method on
child nodes before combining results. This is the standard tree-traversal
pattern for AST evaluation.

---

### Step 2 — Lexer

**File:** `src/prop_evaluator/components/lexica.py`

**Thinking:** The lexer's job is simple: break the raw input string into a
stream of tokens that the parser can consume. For this language there are only 4
token types.

**Tokens defined:**

| Token   | Pattern | Notes            |
| ------- | ------- | ---------------- |
| `AND`   | `∧`     | Unicode U+2227   |
| `OR`    | `∨`     | Unicode U+2228   |
| `TRUE`  | `t`     | Single character |
| `FALSE` | `f`     | Single character |

Whitespace (spaces, tabs, newlines) is silently ignored via `ignore = " \t\n"`.

The operators are defined **before** the atoms in the file. In `sly`,
string-type token rules are matched in order of **decreasing length** (longer
patterns first), but keeping operators first is cleaner and avoids any potential
ambiguity with future tokens.

---

### Step 3 — Parser

**File:** `src/prop_evaluator/components/parser.py`

**Thinking:** The parser receives the token stream from the lexer and constructs
the AST. Each grammar rule's action creates and returns an AST node rather than
evaluating immediately.

**Grammar rules → AST construction:**

```python
@_("expr OR expr")
def expr(self, p) -> PropExpression:
    return PropOr(p.expr0, p.expr1)   # build a node, don't evaluate

@_("expr AND expr")
def expr(self, p) -> PropExpression:
    return PropAnd(p.expr0, p.expr1)

@_("TRUE")
def expr(self, p) -> PropExpression:
    return PropTrue()

@_("FALSE")
def expr(self, p) -> PropExpression:
    return PropFalse()
```

The precedence declaration:

```python
precedence = (
    ("left", OR),    # ← lower
    ("left", AND),   # ← higher
)
```

...resolves all shift/reduce conflicts automatically. `sly`/`yacc`-style parsers
consult this table when they encounter an ambiguity and choose the action that
respects the declared precedence.

**Integration with the lexer:**

```python
tokens = PropLogicLexer.tokens
```

This single line pulls the token set from the lexer into the parser, the same
pattern used in the starter project.

### Step 4 — GUI

**File:** `src/prop_evaluator/main.py`

**Thinking:** The GUI layout is defined in `components/main.ui` using Qt
Designer's XML format, then compiled to `components/ui.py` via `pyside6-uic`.
`main.py` loads this compiled class (`Ui_MainWindow`) and binds each button to
its handler — the same pattern used in the starter project's `src/example/`.

**Layout:**

```
[ Input: _________________________________ ]

[ t ]  [ f ]  [ ∧ ]  [ ∨ ]  [ Clear ]  [ Evaluate = ]

─────────────────────────────────────────
Truth Value:   t
Prefix:        ∨ t ∧ f f
```

**Wiring:**

1. Each atom/operator button appends its character to the input field.
2. `Clear` resets the input and both output labels.
3. `Evaluate` (or pressing Enter) runs the full pipeline:
   ```
   input string
       → PropLogicLexer.tokenize()
       → PropLogicParser.parse()
       → AST root node
       → root.run()    → truth value label
       → root.prefix() → prefix label
   ```

## Method of Translation and Integration

The translation from infix expression to both outputs follows the **AST
interpreter** pattern:

```
Input text  ──[Lexer]──▶  Token stream
                              │
                          [Parser]
                              │
                           AST root
                          /        \
                   [run()]        [prefix()]
                      │                │
               bool result       prefix string
                      │                │
               truth label      prefix label
```

**Key design decision — deferred evaluation:** The parser never computes a
value. It only _builds_ the tree. The two outputs are produced by two
independent traversals of the same tree after parsing completes. This cleanly
separates:

- **Parsing** (syntax) from **evaluation** (semantics)
- The two different semantic interpretations (truth value vs. prefix notation)

## Example Walkthrough

Input: `t ∨ f ∧ f`

**Lexing:**

```
TRUE  OR  FALSE  AND  FALSE
```

**Parsing** (respecting AND > OR precedence):

```
PropOr(
    PropTrue(),
    PropAnd(PropFalse(), PropFalse())
)
```

**`run()` traversal:**

```
PropOr.run()
  → PropTrue().run()              = True
  → PropAnd(f, f).run()
      → PropFalse().run()         = False
      → PropFalse().run()         = False
      → False and False           = False
  → True or False                 = True   ✓
```

**`prefix()` traversal:**

```
PropOr.prefix()
  → "∨ " + PropTrue().prefix() + " " + PropAnd(f,f).prefix()
  → "∨ " + "t"                  + " " + "∧ f f"
  → "∨ t ∧ f f"                                              ✓
```
